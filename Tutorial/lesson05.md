# 第 5 课：总入口 `LLM` 与 `LLMEngine`

## 本课在整套课程中的位置

第 3 课你从示例程序看到了外部 API。

第 4 课你又把输入参数分成了：

- 引擎级配置
- 请求级采样配置

现在我们终于可以进入系统总入口，回答这个问题：

> 当用户调用 `LLM(...)` 和 `generate(...)` 时，系统内部第一层到底发生了什么？

这就是第 5 课的核心。

这一课非常重要，因为它是整个项目的第一个真正中枢：

- 上游接外部 API
- 下游接调度器和模型执行器

如果你能吃透 `LLMEngine`，后面所有模块都会开始“连起来”。

## 本课目标

学完这一课后，你应该能够：

1. 解释为什么 `LLM` 几乎只是一个门面类
2. 说清楚 `LLMEngine.__init__()` 做了哪些关键初始化
3. 说清楚 `add_request`、`step`、`generate` 分别承担什么职责
4. 用自己的话描述一次生成请求如何进入运行时主循环

## 本课涉及文件

- `nanovllm/llm.py`
- `nanovllm/engine/llm_engine.py`
- `nanovllm/config.py`
- `nanovllm/sampling_params.py`

## 先给出结论

这一课最重要的结论有 5 个：

1. `LLM` 本身几乎不提供逻辑，它只是把对外名字收敛成一个更友好的入口
2. 真正的协调者是 `LLMEngine`
3. `LLMEngine.__init__()` 负责把配置、tokenizer、scheduler、model runner 串起来
4. `generate()` 不是“一次调用一次大前向”，而是一个持续调用 `step()` 的循环
5. `LLMEngine` 的核心价值不是算模型，而是协调整个推理流程

## 一、先看 `LLM`：为什么它几乎是空的

`nanovllm/llm.py` 里只有很少代码：

- `class LLM(LLMEngine): pass`

这说明作者在对外 API 设计上做了一个很直接的选择：

- 对外暴露一个简单、短、符合直觉的类名：`LLM`
- 真正逻辑全部放进 `LLMEngine`

这是一种很常见的工程做法。

它的意义在于：

- 用户代码更自然
- 内部逻辑命名仍然保持清晰

所以你要把 `LLM` 理解成：

> 对外友好名称

把 `LLMEngine` 理解成：

> 实际运行时协调器

## 二、`LLMEngine` 在整个系统里的位置

如果你把整个项目简化成几层：

- 外部调用层
- 运行时组织层
- 执行层
- 算子层

那么 `LLMEngine` 正好站在：

- 外部调用层和运行时组织层之间

它一头连接：

- 用户传入的 prompt
- `SamplingParams`

另一头连接：

- `Scheduler`
- `ModelRunner`
- tokenizer

所以你可以把它理解成：

> 这是用户请求正式进入推理引擎后的第一站。

## 三、`__init__()`：系统实例化时到底做了什么

这一段是本课最关键的部分。

`LLMEngine.__init__()` 大致做了 7 件事。

### 1. 从 kwargs 中筛出真正属于 `Config` 的字段

它先拿到 `Config` dataclass 的字段集合，然后从传入参数里只保留这些字段。

这一步的意义是：

- 对外入口允许使用 `kwargs`
- 但内部不希望所有参数都散落处理
- 所以先集中收敛成一个 `Config`

这说明 `Config` 是系统初始化的统一协议。

### 2. 构造 `Config`

这一点你在第 4 课已经熟悉了。

这里的关键不是“创建对象”本身，而是：

- 从这一刻开始，系统边界已经被固定下来

比如：

- 模型路径
- 并行度
- 是否 eager
- batch 上限

这些都会被固化到引擎实例中。

### 3. 根据并行度决定是否拉起额外进程

这一段非常有信号量：

- 如果 `tensor_parallel_size > 1`
- 会启动额外 `Process`
- 每个额外进程都运行 `ModelRunner`

这说明：

- 多卡并行并不是后面某个模块偶然打开的
- 而是在引擎初始化时就决定了系统拓扑

所以 `LLMEngine` 不只是“调用模型”，它还负责：

- 建立执行进程结构

### 4. 创建 rank 0 的 `ModelRunner`

即使单卡时，也会创建：

- `self.model_runner = ModelRunner(config, 0, self.events)`

这说明 `ModelRunner` 是执行层主入口，而 `LLMEngine` 不自己负责真正模型前向。

它们的分工可以先粗略记成：

- `LLMEngine` 负责组织
- `ModelRunner` 负责执行

### 5. 创建 tokenizer

这里有一个很重要的设计：

- tokenizer 归 `LLMEngine` 持有

这意味着：

- 文本 prompt 的编码
- 结果 token 的解码

都由这一层完成。

所以 `LLMEngine` 不是纯调度器，它仍然保留了“文本接口适配层”的责任。

### 6. 回填 EOS 到 `Config`

这里你会看到：

- `config.eos = self.tokenizer.eos_token_id`

这正好对应第 4 课讲过的内容：

- `Config` 有些字段是初始化后补上的

这里的逻辑非常自然：

- `Config` 先创建
- tokenizer 初始化完成后
- 系统再把真正的 `eos` 补进去

### 7. 创建 `Scheduler`

到这里，系统运行时最核心的两个对象已经齐了：

- `ModelRunner`
- `Scheduler`

从这一刻开始，系统已经具备：

- 调度请求
- 执行模型

的完整框架了。

## 四、为什么 `atexit.register(self.exit)` 很重要

这一行很容易被忽略，但其实有很强的工程意味。

它说明作者知道系统在运行结束时需要：

- 通知 worker 退出
- 回收进程
- 清理资源

也就是说，`LLMEngine` 从初始化那一刻起，不仅负责“怎么启动”，也负责“怎么善后”。

所以你读这种系统代码时，要有一个意识：

- 生命周期管理本身就是入口类的职责之一

## 五、`add_request()`：请求第一次被系统接住

这可能是整份文件里最容易被低估的方法。

它做了两件事：

### 1. 把字符串 prompt 变成 token ids

如果输入是字符串：

- 就通过 tokenizer 编码

这说明：

- 文本接口和内部 token 接口在这里汇合

### 2. 构造 `Sequence`

然后系统把这个请求包装成：

- `Sequence(prompt, sampling_params)`

这一步非常重要，因为它意味着：

> 从这一刻开始，请求不再是“用户输入”，而是“调度系统内部对象”。

也就是说，`add_request()` 是外部世界和内部世界的边界。

## 六、`step()`：整个引擎的最小推进单位

`step()` 是本课必须看懂的方法。

它的逻辑非常紧凑，但恰好把系统主线压缩成了 4 步：

1. `scheduler.schedule()` 决定当前执行哪些序列，以及当前是 prefill 还是 decode
2. `model_runner.call("run", seqs, is_prefill)` 执行本轮模型计算
3. `scheduler.postprocess(...)` 把新 token 回填给序列，并判断是否结束
4. 汇总已经完成的输出

这 4 步几乎就是整套运行时的骨架。

所以你可以把 `step()` 理解成：

> 引擎时钟跳动一次，系统就向前推进一轮。

后面你如果把别的模块看迷糊了，就回到 `step()`。

## 七、`num_tokens` 为什么 decode 阶段是负数

`step()` 里有一个很有意思的小设计：

- prefill 阶段返回正的 token 数
- decode 阶段返回负的序列数

这个设计不是数学意义上的“负 token”，而是为了：

- 在进度条里区分两种吞吐统计

后面 `generate()` 里会用它来分别更新：

- prefill throughput
- decode throughput

这也是一个很典型的工程细节：

- 业务逻辑和观测逻辑是一起设计的

## 八、`is_finished()`：为什么入口层也关心完成状态

这个方法表面上只是：

- 返回 `scheduler.is_finished()`

但它表达了一层很清晰的封装关系：

- 运行结束条件真正由 `Scheduler` 持有
- 对外却通过 `LLMEngine` 暴露

这说明作者不希望用户代码越过入口层直接操作调度器。

所以你要注意：

- `LLMEngine` 不负责自己判断一切
- 但它负责对外统一呈现系统状态

## 九、`generate()`：系统主循环真正在哪里

很多人第一次看推理代码，会以为核心在模型前向。

但在这个项目里，真正的“系统主循环”在：

- `generate()`

它大致做了这些事情：

1. 处理进度条
2. 统一 `sampling_params` 的输入形式
3. 把每个 prompt 加入调度系统
4. 在 `while not self.is_finished()` 中反复调用 `step()`
5. 收集已完成输出
6. 最后把 token ids 解码成人类可读文本

所以这一课最关键的一句话是：

> `generate()` 不是一次 forward，而是一个持续驱动系统前进的循环。

## 十、为什么 `generate()` 要支持单个或多个 `SamplingParams`

这里有一个非常工程化的处理：

- 如果传入的是单个 `SamplingParams`
- 就复制成和 prompts 等长的列表

这说明作者在 API 设计上兼顾了：

- 简单场景易用
- 复杂场景可扩展

也就是说：

- 你可以所有请求共用一套采样参数
- 也可以每个请求一套不同参数

这对于批处理推理非常自然。

## 十一、输出为什么先按 `seq_id` 聚合再排序

这一点很值得注意。

因为调度系统内部：

- 多个请求可能在不同轮次完成

但对外返回结果时，用户通常希望：

- 输出顺序和输入顺序对应

所以 `generate()` 先用 `seq_id` 存起来，最后再排序。

这说明 `LLMEngine` 还承担了一个责任：

- 把内部异步/乱序完成的结果整理成外部友好的稳定顺序

## 十二、`LLMEngine` 负责什么，不负责什么

这节课学到这里，你应该有一个很清晰的分工图。

### `LLMEngine` 负责

- 配置收敛
- tokenizer 接入
- 运行时组件初始化
- 请求入队
- 主循环推进
- 输出整理
- 生命周期管理

### `LLMEngine` 不负责

- 真正的注意力计算
- KV cache 物理分配
- 调度细节判断
- 权重分片加载细节

这些事分别交给了别的模块。

所以它本质上是：

> 系统总协调器，而不是算子实现者。

## 十三、本课最容易出现的误解

### 误解 1：`LLM` 很重要，因为它是对外入口

不准确。

真正重要的是 `LLMEngine`，`LLM` 只是名字更友好的包装。

### 误解 2：`generate()` 里主要做模型前向

不对。

它主要做的是：

- 组织循环
- 推进系统
- 汇总结果

### 误解 3：请求一进入系统就立刻出结果

不对。

请求会先变成 `Sequence`，然后交由调度器逐轮推进。

### 误解 4：入口层不重要，真正重要的是底层算子

不对。

如果没有入口层的统一协调，底层再快也无法组成可用系统。
