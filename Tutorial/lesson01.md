# 第 1 课：项目定位与学习地图

## 本课在整套课程中的位置

这是 30 节课程里的第 1 课，作用不是直接带你看实现细节，而是先建立一个正确的学习框架。

如果这一步没做好，后面会很容易出现两种问题：

- 把注意力放到不重要的细节上，比如先钻 Triton kernel，却没理解调度器为什么存在
- 用错误预期去看项目，比如把它当成通用训练框架，或者把它当成完整生产级 serving 系统

所以第 1 课的目标只有一个：先回答“这个项目是什么，以及应该如何学习它”。

## 本课目标

学完这一课后，你应该能够：

1. 用自己的话说明 nano-vllm 的定位
2. 说清楚它重点实现了什么，不重点实现什么
3. 知道这套代码最值得学习的主线是什么
4. 形成一个后续阅读顺序，而不是盲目跳文件

## 本课涉及文件

- `README.md`
- `pyproject.toml`

这两个文件的信息量其实很大，因为它们会直接告诉你：

- 作者希望你把项目理解成什么
- 这个项目依赖哪些技术栈
- 它面向的运行环境是什么
- 后面课程里哪些主题是主角

## 先给出结论

这个仓库不是一个“从零实现大模型”的项目，而是一个“用尽量少的代码复现 vLLM 风格推理引擎核心思路”的项目。

请你先牢牢记住这一句：

> 你在这里主要学习的是推理时的运行时系统设计，而不是模型训练，也不是通用深度学习框架。

换句话说，这个项目真正有价值的地方，不在于它写了一个 Qwen3，而在于它把下面这些运行时问题压缩成了一个比较容易读懂的代码库：

- 请求如何排队和调度
- 为什么要区分 prefill 和 decode
- KV cache 为什么需要分页管理
- prefix cache 为什么能省时间
- 模型执行层如何接收调度结果
- 多卡和优化路径如何嵌入到执行流程中

## 一、从 README 提取项目定位

先看 `README.md` 里最关键的几句话：

- 这是一个 “A lightweight vLLM implementation built from scratch”
- 代码目标之一是 “Readable codebase”
- 它强调的能力包括：
  - Fast offline inference
  - Prefix caching
  - Tensor Parallelism
  - Torch compilation
  - CUDA graph

从这几句里，你应该提炼出三个判断。

### 1. 它是 vLLM 风格，不是 Hugging Face 风格

Hugging Face `model.generate()` 的学习路径通常是：

- 先看模型结构
- 再看解码策略
- 最后再看缓存和优化

而 vLLM 风格的学习路径应该反过来：

- 先看运行时主线
- 再看请求调度
- 再看 KV cache 和 prefix cache
- 最后再看模型层如何配合运行时

也就是说，这个项目的中心不是“模型”，而是“推理引擎”。

### 2. 它是 lightweight implementation

这意味着作者不是在追求“覆盖一切复杂场景”，而是在追求：

- 保留关键机制
- 去掉大量工程外壳
- 让学习者能在较小代码量里看到本质

所以你读这个仓库时，应该抱着这样的期待：

- 学核心机制，非常合适
- 学生产系统全貌，不够

### 3. 它是 readable codebase

这说明这个项目的一个重要价值就是“适合学习”。

很多生产级项目的问题不是思路不清，而是工程规模太大，导致你第一次读根本无法抓主线。这个仓库的优势恰好相反：

- 代码量小
- 结构集中
- 关键逻辑没有被太多外围系统稀释

这也是为什么它非常适合做课程化学习。

## 二、从依赖文件判断学习重点

再看 `pyproject.toml` 里的依赖：

- `torch`
- `triton`
- `transformers`
- `flash-attn`
- `xxhash`

这 5 个依赖几乎已经把整个课程主线暴露出来了。

### 1. `torch`

说明基础执行框架仍然是 PyTorch。

你后续会看到：

- 模型定义是 PyTorch Module
- 张量准备是 PyTorch
- 多卡通信是 `torch.distributed`
- 一些优化用了 `torch.compile`

### 2. `transformers`

说明这个项目并不是自己发明模型配置和 tokenizer 生态，而是复用 Hugging Face 的模型定义信息。

后续你会看到：

- 用 `AutoConfig` 读取模型结构信息
- 用 `AutoTokenizer` 做文本编码和解码

所以这里的重点不是“自己造 tokenizer”，而是“把已有模型权重接到自定义 runtime 上”。

### 3. `flash-attn`

这直接说明：注意力计算不是简单地手写矩阵乘法，而是用了高性能实现。

你后面应该重点关注：

- prefill 阶段如何走变长 attention
- decode 阶段如何结合 kv cache 做 attention

### 4. `triton`

这意味着项目里会有自定义 GPU kernel，但你不要因此误判重点。

这里一个非常重要的学习原则是：

> 在没有吃透调度、cache 和数据组织之前，不要先钻 Triton kernel。

Triton 在这个项目里重要，但不是第一优先级。第一优先级永远是系统主线。

### 5. `xxhash`

这个依赖非常有信号量，因为它往往意味着：

- 某处需要快速哈希
- 哈希大概率和 cache 复用有关

在这个项目里，它确实对应 prefix cache 的块级哈希设计。这一点会在后面课程详细展开。

## 三、这个项目重点实现了什么

从 `README.md` 和依赖文件综合判断，这个项目最值得学习的内容有 6 个。

### 1. 请求级推理流程

也就是：

- 用户传入 prompt
- 系统把它包装成内部请求对象
- 调度器安排它进入 prefill 或 decode
- 模型执行层根据调度结果运行
- 结果回填并决定是否结束

这条主线会贯穿整个项目。

### 2. prefill / decode 双阶段设计

这是现代 LLM 推理引擎最重要的分层之一。

你不能把生成过程简单理解成“每次都 forward 一下模型”。这个项目会让你看到：

- 首次处理 prompt 和后续逐 token 生成，是两种完全不同的工作负载
- 这两种工作负载必须在 batch 组织、cache 使用和性能优化上分开处理

### 3. paged KV cache

这不是一个“附带优化”，而是整个系统能高效扩展的关键机制。

如果不按 block 管理 KV cache，很多现实问题会变得非常麻烦，比如：

- 变长请求并发
- 动态扩容
- cache 复用
- 显存回收

### 4. prefix cache

这是这个项目很值得学的点，因为它把“相同前缀请求共享已有计算结果”这件事做得非常清晰。

你需要逐步理解：

- 为什么要在 block 粒度做
- 为什么完整 block 才能稳定复用
- 为什么需要哈希和引用计数

### 5. Tensor Parallel

这个项目虽然不是重型分布式系统，但它把张量并行的核心思路保留下来了。

这意味着后面你不仅会学到“单卡上怎么跑”，还会学到：

- 权重怎么切
- embedding 和 lm head 怎么切
- 不同线性层为什么用不同分片方式

### 6. CUDA Graph / torch.compile / 高性能注意力

这些优化不是单纯为了“更快”，而是为了让你看到现代推理引擎常见的优化方向。

后面你应该带着一个问题去看：

> 每个优化到底解决了什么瓶颈，它为什么能成立？

## 四、这个项目不重点解决什么

为了避免学偏，你也要知道这个仓库没有把重点放在哪些方向。

### 1. 不重点讲训练

这个仓库几乎不涉及：

- 训练循环
- 反向传播优化
- optimizer
- 数据集处理

所以不要把它当训练框架读。

### 2. 不重点讲完整服务系统

这里没有把重点放在：

- HTTP server
- 请求接入层
- 分布式调度中心
- 租户隔离
- 大规模生产监控

所以它更像“推理引擎核心机制教学版”，不是完整在线服务产品。

### 3. 不重点支持大量模型架构

当前代码重点围绕 Qwen3 路径展开，这会带来两个结果：

- 好处是结构集中，学习成本低
- 限制是你学到的是运行时核心机制，不是全模型生态支持

### 4. 不重点兼容所有硬件路线

从依赖上就能看出来，这个项目主要是：

- CUDA
- NCCL
- FlashAttention
- Triton

它不是面向 CPU/MPS 的通用教程。

## 五、和另外两类项目的区别

为了帮你建立正确参照系，建议你把 nano-vllm 和下面两类项目区分开。

### 1. 和 Hugging Face Transformers 的区别

Hugging Face 更偏：

- 模型定义
- 权重加载
- 通用接口
- 各种模型架构支持

nano-vllm 更偏：

- 推理运行时
- 批处理调度
- cache 管理
- 高性能执行路径

一句话总结：

- Hugging Face 更像“模型框架”
- nano-vllm 更像“推理引擎教学版”

### 2. 和真正的 vLLM 仓库的区别

真正的 vLLM 更偏：

- 完整生产级能力
- 更复杂的特性支持
- 更大规模的工程系统
- 更多边界情况与性能路径

nano-vllm 更偏：

- 抽取核心思想
- 用较少代码保留关键机制
- 让学习者快速抓住主线

一句话总结：

- vLLM 是完整工业实现
- nano-vllm 是适合深入学习核心机制的缩小版

## 六、正确的学习主线是什么

这门课最重要的输出，不是让你记住仓库介绍，而是建立一条正确的阅读主线。

后面所有课程都应该围绕下面这条线推进：

1. 外部 API 怎么调用
2. 请求如何变成内部 `Sequence`
3. 调度器如何在 prefill 和 decode 之间安排执行
4. block manager 如何分配和复用 KV cache
5. model runner 如何把调度结果转换成模型输入
6. attention 如何真正读写 KV cache
7. 采样器如何产出下一个 token
8. 优化路径如何嵌入这条主线

如果你以后读代码时迷路了，就回到这 8 步。

## 七、第 1 课之后，你暂时不要做什么

这部分很重要。

第 1 课学完之后，先不要：

- 立刻钻 `Triton` kernel
- 立刻看多卡共享内存细节
- 立刻纠结每一个 PyTorch API
- 立刻尝试支持新模型

因为这些都不是最先决定你能否读懂这个项目的东西。

第 1 课之后你真正应该具备的是：

- 明白项目类型
- 明白后续重点
- 明白阅读顺序
